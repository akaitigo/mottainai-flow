# PRD: mottainai-flow

## 概要

スーパー、農家、食品メーカーの余剰在庫をフードバンクや子ども食堂へ最適配車する運用プラットフォーム。
「もったいない」を掲示板ではなく**配送最適化問題**として解く。

## 背景・課題

- 日本のフードロスは年間約523万トン（農水省2023年推計）
- フードバンク団体の多くは手動で電話/メール調整しており、マッチングと配車に人的コストがかかる
- 既存のフードシェアアプリは「掲示板型」が主で、配送ルート最適化まで踏み込んでいない

## ゴール

余剰食品の提供者と受取者を結び、集荷ルートを自動提案することで、配送コストを最小化しつつ食品ロスを削減する。

## ユーザー

| ロール | 説明 |
|--------|------|
| 提供者 | スーパー、農家、食品メーカー。余剰在庫を登録する |
| 受取者 | フードバンク、子ども食堂、福祉施設。必要な食品を登録する |
| 配送者 | ボランティアドライバー。提案されたルートに基づき集荷・配送する |
| 管理者 | プラットフォーム運営。マッチング監視、データ分析 |

## MVP機能

### F1: 提供・受取在庫の登録
- [x] 提供者が余剰食品を登録（品目、数量、消費期限、受取可能時間帯、住所）
- [x] 受取者が必要な食品を登録（品目カテゴリ、希望数量、受取可能時間帯、住所）
- [x] 住所からジオコーディングし PostGIS に保存

### F2: 集荷ルート自動提案
- [x] 提供・受取のマッチングを距離・時間帯・品目で自動実行
- [x] OR-Tools で最適集荷ルート（VRP: Vehicle Routing Problem）を計算
- [x] 配送者にルート提案を通知

### F3: 受領記録とトレーサビリティ管理
- [x] 配送者が受領確認を記録（写真、数量、状態）
- [x] 提供者・受取者がステータスを追跡
- [x] トレーサビリティレポートの生成

## 技術スタック

| レイヤー | 技術 |
|----------|------|
| バックエンド API | Kotlin + Quarkus (gRPC) |
| ルート最適化サービス | Go + OR-Tools |
| フロントエンド | TypeScript + React |
| データベース | PostgreSQL + PostGIS |
| 通信 | gRPC (サービス間), REST (フロントエンド向け) |
| インフラ | Docker Compose (開発), GCP Cloud Run (本番想定) |

## 非機能要件

- レスポンス: API 95%ile < 500ms
- 同時配送ルート計算: 100件/分
- データ保持: 受領記録は3年間保存
- セキュリティ: 個人情報（住所）は暗号化保存

## 成功指標

- MVP で 10 提供者、5 受取者が登録し、月 20 回の配送が成立すること
- 手動調整と比較して配送ルート提案の所要時間を 80% 削減

## スコープ外（MVP）

- 決済機能
- リアルタイム配送追跡（GPS）
- モバイルネイティブアプリ（PWA で代替）
- 多言語対応
