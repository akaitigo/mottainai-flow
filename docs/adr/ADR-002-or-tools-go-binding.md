# ADR-002: OR-Tools Go バインディング選定

**日付**: 2026-03-30
**ステータス**: 承認済み
**決定者**: 開発チーム

## コンテキスト

mottainai-flow のルート最適化サービス（F2）では、Vehicle Routing Problem (VRP) を解くために Google OR-Tools を使用する。Go サービスから OR-Tools を呼び出す方法を選定する必要がある。

## 検討した選択肢

### 選択肢A: CGO バインディング（or-tools C++ ライブラリ直接呼出し）

**メリット**: OR-Tools のフル機能を利用可能。最高のパフォーマンス。
**デメリット**: CGO のビルド環境構築が複雑（C++ コンパイラ、OR-Tools ライブラリのリンク）。クロスコンパイルが困難。Docker イメージサイズが肥大化。依存関係の管理が煩雑。

### 選択肢B: gRPC 経由で Python/Java OR-Tools サーバーを呼び出し

**メリット**: OR-Tools の公式サポートが充実した言語を使える。
**デメリット**: サービス間通信のレイテンシ。追加のサービス管理が必要。Go サービスの存在意義が薄れる。

### 選択肢C: Pure Go による VRP ソルバー自前実装

**メリット**: 依存ゼロ。ビルド・デプロイが簡単。Go エコシステムに閉じた開発。テストが容易。
**デメリット**: OR-Tools と同等の最適化性能は得られない。初期実装コストは高いが、MVP スコープでは許容範囲。
**実装方針**: Nearest Neighbor + 2-opt 改善法で十分な品質を確保。

## 決定

### 選択肢C を採用: Pure Go VRP ソルバー

MVP スコープでは 100 件/分のスループット要件に対して、Pure Go の Nearest Neighbor + 2-opt アルゴリズムで十分対応可能。

理由:
1. **ビルド・デプロイの簡潔さ**: CGO なしで `go build` 一発。Docker マルチステージビルドも簡単
2. **テスト容易性**: 既知の最適解を持つテストケースで検証しやすい
3. **MVP に十分な精度**: 小〜中規模の VRP（拠点数 < 50）では Nearest Neighbor + 2-opt で実用的な解が得られる
4. **将来の拡張性**: インターフェースを抽象化し、後から OR-Tools CGO バインディングや外部サービスに差し替え可能

### VRP アルゴリズム設計

1. **距離行列計算**: Haversine 公式による地球表面距離
2. **初期解生成**: Nearest Neighbor ヒューリスティック（最も近い未訪問拠点を選択）
3. **解の改善**: 2-opt（ルート内の辺交換で短縮化）
4. **時間枠制約**: フィルタリング（時間枠外の拠点を除外）+ ペナルティ付きスコア

## 結果

**ポジティブ**:
- ゼロ外部依存。Go標準ライブラリのみで実装可能
- Docker イメージが軽量（CGO 不要）
- テストが確定的で再現性が高い

**ネガティブ/トレードオフ**:
- 大規模 VRP（拠点数 > 100）では OR-Tools に劣る最適性
- メタヒューリスティクス（Simulated Annealing 等）への拡張には追加実装が必要

**リスク**:
- MVP 後にスケール要件が変わった場合、ソルバーの差し替えが必要（インターフェース抽象化で軽減）
